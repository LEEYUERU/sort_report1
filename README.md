## 氣泡排序法 (Bubble Sort)

* **簡述原理：** 
  氣泡排序法透過重複走訪陣列，每次比較相鄰的兩個元素，若順序錯誤則進行交換。走訪過程會讓較大的元素如同「氣泡」般逐漸浮到陣列的尾端。每完成一輪，未排序區間的最大值就會就位。
* **複雜度分析：**
  * **最佳時間複雜度：** `O(n)`（陣列已排序時，加入提前結束的優化機制）
  * **最差/平均時間複雜度：** `O(n^2)`（陣列為反序時）
  * **空間複雜度：** `O(1)`（原地排序）
* **模擬實作內容：**
  在視覺化頁面中，透過動態長條圖展示陣列狀態。當兩個元素正在進行**比較或交換**時，柱狀體會顯示為「紅色」並同步高亮旁邊的演算法偽代碼；確認**已排序**的元素則會轉為「綠色」，讓每輪的氣泡上浮過程一目了然。

## 選擇排序法 (Selection Sort)

* **簡述原理：** 
  將陣列分為「已排序」與「未排序」兩部分。每次在未排序的區間內掃描，尋找出最小值，然後將該最小值與未排序區間的第一個元素進行交換。重複此步驟直到所有元素皆排序完畢。
* **複雜度分析：**
  * **最佳/最差/平均時間複雜度：** `O(n^2)`（無論初始狀態為何，皆需掃描剩餘的所有元素來確認最小值）
  * **空間複雜度：** `O(1)`（原地排序）
* **模擬實作內容：**
  實作中特別強調了「尋找最小值」的過程。掃描比對中的游標會以「紅色」顯示，而**當前找到的最小值**會以「紫色」標記並微幅上浮。確認交換後，加入已排序區的元素會轉為「綠色」，配合右側偽代碼的動態跳轉，清楚呈現選擇與交換的邏輯。

## 插入排序法 (Insertion Sort)

* **簡述原理：** 
  類似於打撲克牌時整理手牌的方式。演算法會逐一取出未排序區間的元素（Key），由後往前與已排序區間的元素進行比較。若掃描到的元素較大，則將其往後平移，直到找到合適的位置將 Key 插入。
* **複雜度分析：**
  * **最佳時間複雜度：** `O(n)`（陣列已完全排序時，不需任何位移）
  * **最差/平均時間複雜度：** `O(n^2)`（陣列為反序時，每次插入都需要移動大量元素）
  * **空間複雜度：** `O(1)`（原地排序）
* **模擬實作內容：**
  動畫以「紫色」標示當前準備插入的元素（Key），在它往前尋找位置時，被它擠開而**向後位移**的元素會亮起「紅色」。這樣的視覺回饋完美還原了「挪出空位並插入」的核心概念。

## 合併排序法 (Merge Sort)

* **簡述原理：** 
  採用分治法（Divide and Conquer）的策略。首先將陣列不斷對半分割，直到每個子陣列只剩下一個元素；接著將相鄰的子陣列兩兩合併，合併時進行排序，直到所有元素合併成一個完整的有序陣列。
* **複雜度分析：**
  * **最佳/最差/平均時間複雜度：** `O(n log n)`（分割過程為 log n 層，每層合併需 O(n) 的時間）
  * **空間複雜度：** `O(n)`（合併時需要配置與原陣列等大的額外暫存空間）
* **模擬實作內容：**
  為了克服合併排序較難視覺化的問題，頁面特別加入了「**局部聚光燈 (Dimming)**」效果。在處理特定的子陣列時，非當前區間的元素會變暗。同時，在長條圖下方加入了數值方格，正在比較的元素會顯示「黃色」，而寫回原陣列時會顯示為「綠色」，藉此具象化遞迴與合併的過程。

## 計數排序法 (Counting Sort)

* **簡述原理：** 
  這是一種非比較型的排序演算法。它藉由找出陣列中的最大值 `k`，建立一個長度為 `k+1` 的計數陣列。掃描原陣列並統計每個數字出現的次數後，再依據計數陣列的順序與次數，依序將數值寫回原陣列中。
* **複雜度分析：**
  * **時間複雜度：** `O(n + k)`（n 為陣列長度，k 為數值範圍。只需走訪陣列常數次，效率極高）
  * **空間複雜度：** `O(k)`（需額外配置長度為 k+1 的計數陣列，受數據範圍限制較大）
* **模擬實作內容：**
  有別於其他排序，本實作在畫面上半部保留原陣列，下半部額外動態生成了「計數陣列」。動畫分為兩個階段：第一階段為「掃描計數」（紅色），可觀察下方計數格子的數字增加；第二階段為「依序寫回」（紫色），清楚展示了不依賴元素互換也能完成排序的特殊機制。

## 綜合比較

| 演算法 | 最佳時間複雜度 | 最差時間複雜度 | 空間複雜度 | 穩定性 (Stability) | 適用場景 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **氣泡排序** | `O(n)` | `O(n^2)` | `O(1)` | 穩定 | 幾乎已排好序的極小數據 |
| **選擇排序** | `O(n^2)` | `O(n^2)` | `O(1)` | 不穩定 | 記憶體寫入成本極高時（交換次數最少） |
| **插入排序** | `O(n)` | `O(n^2)` | `O(1)` | 穩定 | 數據量小，或資料大致已排序的狀況 |
| **合併排序** | `O(n log n)` | `O(n log n)` | `O(n)` | 穩定 | 資料量大且要求穩定排序結果 |
| **計數排序** | `O(n + k)` | `O(n + k)` | `O(k)` | 穩定 | 數據範圍集中且非負整數的場景 |

*(註：穩定性是指相等的元素在排序後，是否能保持原本的相對順序。)*

## 學習心得

在這次的專題實作中，我不僅複習了各項資料結構與演算法的底層邏輯，更獲得了將「抽象代碼」轉化為「具體視覺介面 (GUI)」的寶貴經驗。

過去在實作演算法時，往往只能依靠終端機介面 (Console) 印出一行行的數字來確認結果。然而，這次透過網頁前端技術開發視覺化工具，我深刻體會到「狀態管理」與「非同步動畫控制」的挑戰。為了讓長條圖的高度變化、顏色切換與偽代碼的高亮能夠精準同步，我必須將演算法的每一步驟（如比較、交換、覆蓋）詳細記錄進狀態陣列中，再利用 `async/await` 進行流暢的播放控制。

在開發諸如「局部聚光燈效果」或是「多重狀態顏色管理（當前尋找的最小值 vs 比較中的元素）」時，我意識到良好的使用者體驗來自於細節的打磨。這個專案不僅提升了我對程式邏輯邊界條件的掌握度，更讓我體會到將嚴肅的計算科學轉化為互動教學工具的成就感。未來面對更複雜的工程問題時，我更能熟練地運用拆解與視覺化的思維來尋求解答。
